**Part A: Create `source/MawaqitServiceDelegate.mc`**

This is a dedicated lightweight background service delegate. It does NOT reuse MawaqitService (research explains why: 6-step chain risks 30-second timeout, singleton doesn't work across process boundaries, would bloat the 28KB glance+background memory budget).

NOTE on D-10: CONTEXT.md D-10 says "Calls MawaqitService.fetchPrayerData() from onTemporalEvent()". Research section "Open Questions #3" explains this is not feasible due to timeout/memory/process-boundary constraints. We satisfy D-10's intent (fetch prayer data in background) with a single-request approach instead.

```monkeyc
import Toybox.System;
import Toybox.Background;
import Toybox.Communications;
import Toybox.Application.Properties;
import Toybox.Lang;

//
// MawaqitServiceDelegate: Lightweight background service for periodic prayer data refresh.
//
// Makes a SINGLE HTTP request to /prayer-times endpoint (well under 30-second timeout
// and 8KB Background.exit() data limit). Does NOT reuse MawaqitService which has a
// 6-step chain unsuitable for background execution.
//
// Annotated (:background) so this code loads in the background process context.
// Background and foreground do NOT share memory -- all data transfer goes through
// Background.exit() -> AppBase.onBackgroundData().
//
(:background)
class MawaqitServiceDelegate extends System.ServiceDelegate {

    function initialize() {
        ServiceDelegate.initialize();
    }

    function onTemporalEvent() as Void {
        var slug = Properties.getValue("mosqueSetting") as String or Null;
        if (slug == null || slug.equals("")) {
            Background.exit(null);
            return;
        }

        var url = "https://mawaqit.naj.ovh/api/v1/" + slug + "/prayer-times";
        Communications.makeWebRequest(
            url,
            null,
            {
                :method => Communications.HTTP_REQUEST_METHOD_GET,
                :responseType => Communications.HTTP_RESPONSE_CONTENT_TYPE_JSON
            },
            method(:onReceive)
        );
    }

    function onReceive(responseCode as Number, data as Dictionary or String or Null) as Void {
        if (responseCode == 200 && data != null) {
            Background.exit(data);
        } else {
            // Silent failure -- next scheduled run will retry.
            // Do not crash or throw; just exit with null.
            Background.exit(null);
        }
    }
}
```

   This is necessary because `getServiceDelegate()` must be findable in the background context (Pitfall 4 from research).

3. **Add `getServiceDelegate()` method** (annotated `(:background)`) inside the class, after `getGlanceView()`:
   ```monkeyc

   function getServiceDelegate() as [System.ServiceDelegate] {
       return [new MawaqitServiceDelegate()];
   }
   ```

4. **Add `onBackgroundData(data)` method** inside the class, after `getServiceDelegate()`:
   ```monkeyc
   function onBackgroundData(data) as Void {
       if (data != null) {
           Storage.setValue("todayTimes", data);
           var info = Gregorian.info(Time.now(), Time.FORMAT_SHORT);
           var dateStr = info.year + "-" + info.month + "-" + info.day;
           Storage.setValue("lastFetchDate", dateStr);
           // lastFetchSlug is set by foreground fetch -- don't overwrite here
           // because background doesn't know if the slug changed since last foreground fetch
       }
       WatchUi.requestUpdate();
   }
   ```

5. **Add temporal event registration** in `getInitialView()`, before the fetch call. Add this block at the beginning of getInitialView():
   ```monkeyc
   // Register background refresh if not already registered (Pitfall 7: avoid duplicate registration)
   var registeredTime = Background.getTemporalEventRegisteredTime();
   if (registeredTime == null) {
       // Once daily refresh (86400 seconds = 24 hours) per D-08
       Background.registerForTemporalEvent(new Time.Duration(86400));
   }
   ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Phone BLE -> Background service | HTTP request goes through phone relay; response may be malformed or missing |
| Background process -> Foreground process | Data passes through Background.exit() with ~8KB limit |
| API response -> Storage | Untrusted JSON from API stored directly |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03 | T (Tampering) | API response in background | accept | Same API endpoint as foreground fetch (Phase 1). HTTPS transport. No auth tokens at risk. Malformed data handled by null checks in PrayerLogic/PrayerDataStore. |
| T-03-04 | D (Denial of Service) | Background exit data size | mitigate | Single /prayer-times response is ~200 bytes (6 key-value pairs), well under 8KB limit. Do not pass large payloads through Background.exit(). |
| T-03-05 | D (Denial of Service) | Background timeout | mitigate | Single HTTP request instead of 6-step chain. If request fails, Background.exit(null) prevents hang. Next scheduled run retries. |
| T-03-06 | I (Information Disclosure) | Mosque slug in Properties | accept | Slug is not sensitive data. Stored in standard app Properties accessible only to this app. |
</threat_model>
